Linux查看服务状态和日志.md

发表于 2026-04-24 更新于 2026-04-26 分类于 技术笔记

前言

在 Linux 上搭建服务的时候，经常会遇到各种各样的故障、异常和告警，为了更好的排错，我整理了以下查看服务状态与日志的基本思路。

systemctl status 查看服务状态

首先，通过 systemctl status 查看服务的当前状态，以及该服务最近几条相关日志摘要。适合作为第一轮判断：服务是否正在运行、是否设置为开机自启、主进程是谁、最近一次启动/停止有没有明显报错，从而判断是否需要继续深挖日志。

先给例子：

核心字段：

字段 含义 关键判断
Loaded unit 文件是否被 systemd 正常加载（loaded）、开机自启动 enabled=开机自启; disabled 反之
Active 当前运行状态 active (running)= 正常运行；inactive (dead)= 已停止；failed= 启动失败
Main PID 主进程 ID 用于后续排查进程资源占用
末尾日志那段 服务最近的报错/运行日志 启动失败时，这里会直接显示核心报错原因

systemctl status 底部展示的几行内容，本质上也是该服务在 journal 中最近的日志摘要，有固定的通用格式和判断规则。
日志的固定字段结构：
[时间戳] [主机名] [产生日志的进程名[PID]]: [日志正文]
它们的关键作用：

  • 时间戳：定位问题发生的时间，梳理时间的先后顺序
  • 主机名：多机器场景下，确认是哪台服务器产生的日志
  • 产生日志的进程名[PID]：核心！区分日志的来源：
    • systemd[1]：系统 1 号进程，服务管理器，记录的是服务的生命周期（启动、停止、重启），属于服务的“外部状态日志”
    • 比如 nginx[917]/mysql[1107]：服务自身的进程，记录的是程序内部的运行情况（配置加载、客户端连接、程序报错等），属于服务的“内部运行日志”
    • 日志正文：告诉你具体发生了什么事，是判断问题的核心

一般来说，想通过 systemctl status 给出的日志来判断问题的严重程度有三个步骤：
一、先看 Active 和日志最后一行，清楚服务的状态

  • 只要 Active 是 active(running)，然后日志也显示 Started xxx，那么中间的警告或报错在当前大概率不是致命问题，但仍虚结合访问验证或进一步查看日志，确认业务功能是否完全正常。
  • Active 是 inactive(dead)，然后日志显示 Failed to start xxx 的话，那就是启动失败。
    二、看日志的来源
  • 比如来源是 systemd[1]：那就说明问题更偏向于服务生命周期和 systemd 视角，比如启动权限不足、端口占用、PID 文件异常、服务文件配置错误等等。
  • 来源是 nginx[917]/mysql[1107]：问题更偏向于程序内部执行细节，比如配置文件语法错误、数据库密码不对、连接数超限、文件读写权限不足等。
    三、通过关键词识别事件类型

    这些关键词适合做快速判断，但不能替代完整日志分析。

日志关键词 含义 状态判断
Starting xxx 开始启动 xxx 服务 流程启动，中性通知
Started xxx 成功启动 xxx 服务 启动完成，服务正常
Stopping xxx 开始停止 xxx 服务 停止流程启动，中性通知
Stopped xxx 成功停止 xxx 服务 停止完成，中性通知
Failed to start xxx 启动 xxx 服务失败 严重错误，服务未启动
Main process exited, code=exited 服务主进程异常退出 严重错误，服务崩溃

这里补充日志的优先级：
emerg(紧急) > alert(警报) > crit(严重) > err(错误) > warn(警告) > notice(通知) > info(信息) > debug(调试)
systemctl status 默认只展示这个 unit 最近的若干条日志摘要。

两种查看日志的方式

刚刚的 systemctl status 只是看服务的状态和几条最近的日志，只是一个概览，要想全面地深挖具体报错的细节，还得靠完整日志。
常见地查看日志的方式有 journalctl 命令和日志文件。

/var/log/ 目录日志文件

这个目录下的日志主要分为两类，一种是系统级的日志文件，一种是由应用自主写入的业务日志。当 journalctl 不够用，或者要看应用访问日志/错误日志时，就需要再去 /var/log/... 对应目录找对应日志。

一般来说，系统服务层（启动失败、异常退出、systemd 报错、资源/权限问题、内核/认证事件）优先选 journalctl。如果问题已经落到应用层，就进一步分析该应用对应的日志文件，例如应用的访问日志、报错日志和慢查询日志。

系统核心的通用日志有：

日志路径 适用系统 日志内容
/var/log/messages 红帽系 系统整体运行日志、绝大多数服务的常规信息
/var/log/syslog Debian 系 对应红帽系的 messages，系统全局日志
/var/log/secure 红帽系 安全认证日志：ssh 登录、sudo 权限、用户变更等
/var/log/auth.log Debian 系 对应红帽系的 secure，认证安全日志
/var/log/cron 全系统通用 定时任务 crontab 的执行日志，排查定时任务失败
/var/log/dmesg 全系统通用 内核启动、硬件驱动相关日志，排查硬件 / 内核问题

常见服务专属日志路径：

服务 标准日志路径 核心日志文件
Nginx /var/log/nginx/ access.log（访问日志）、error.log（错误日志）
Apache/httpd 红帽系 /var/log/httpd/；Debian 系 /var/log/apache2/ 访问日志、错误日志
MySQL/MariaDB /var/log/mariadb/、/var/log/mysql/ 错误日志、慢查询日志
PostgreSQL /var/log/postgresql/ 实例运行日志
Docker 比较特殊，容器日志一般在 /var/lib/docker/containers，服务运行日志用 journalctl -u docker 容器运行日志，也可用 journalctl -u docker
