Linux排障：进程、端口与下一步判断

发表于 2026-04-29 分类于 技术笔记

ps aux 的作用是什么

ps 全称是 process status（进程状态），用于查看当前系统的进程信息。
aux 是命令参数，但没有前置横杠，这是 BSD 系统的语法。

1 UNIX options, which may be grouped and must be preceded by a dash.
2 BSD options, which may be grouped and must not be used with a dash.
3 GNU long options, which are preceded by two dashes.
—— man 文档。（dash 就是破折号）

  • a 与 x 经常一起使用：a （所有用户的进程）参数搭配 x （没有控制终端的进程）可以列出所有进程。
  • u：以 user-oriented 格式输出，面向用户的详细格式输出。
    示例：
字段 核心含义
USER 进程的运行用户
PID 进程唯一 ID，核心标识
%CPU CPU 使用率
%MEM 占用的物理内存百分比
VSZ 占用的虚拟内存大小 KB
RSS 占用的物理内存大小 KB
TTY 表示进程关联的终端，? 代表与终端无关。pts/0 表示关联在一个伪终端上，比如 SSH 登录或终端模拟器会话
STAT 进程目前的状态。R：正在运行；S：可中断睡眠；D：不可中断的睡眠；T：已停止或被跟踪；Z：僵尸进程
START 进程触发启动的时间
TIME 进程实际使用 CPU 运行的时间
COMMAND 进程的实际命令是什么

ss 和 lsof 的区别是什么

ss：和 netstat 的功能差不多，目前算是 netstat 的替代方案了。主要用来查看 socket 连接与端口监听状态，排查“服务是否真的在监听某个端口”时非常常用。
lsof：list open files，可以打开查看所有文件句柄，Linux 万物即文件，也就是说，普通文件、目录、设备、 Socket 等等都可以查看。相对于 ss 命令，lsof 不仅可以查看端口占用情况，也可以查看进程打开了哪些文件 (端口)。

查完进程和端口后 ，下一步通常要干嘛

场景 1：进程不存在
比如：

  • ps aux | grep nginx 看不到 nginx 主进程。
  • ss -lntp | grep :80 也看不到 80 端口监听
    这通常说明：服务根本没起来，或者已经异常退出了。

下一步应当：

  1. 用 systemctl status nginx 查看服务当前状态；
  2. 用 journalctl -u nginx -n 50 看最近的日志；
  3. 检查配置文件、权限、依赖目录是否有问题；
  4. 修复后重新启动服务，再重复验证进程和端口。

场景 2：进程存在，但端口没监听
比如：

  • ps aux | grep nginx 能看到 nginx 进程。
  • ss -lntp | grep :80 看不到 80 端口监听
    这通常说明：
  • 服务虽然有进程，但没有真正完成监听；
  • 或者监听配置写错了；
  • 或者进程卡在启动后的异常状态。

下一步应当：

  1. 看配置文件是否正确配置了对应端口；
  2. 看日志里有没有 bind failed、permission denied、address already in use 等报错；
  3. 检查服务是否只监听在 127.0.0.1，而不是 0.0.0.0；
  4. 检查是否被其他进程抢占了目标端口。

场景 3：进程和端口正常，但访问还是失败
这通常说明：问题可能已经不在“服务是否启动”这一层，而在网络路径、访问控制或应用内部。

下一步应当：

  1. 先本机访问：curl 127.0.0.1:端口 或 curl localhost；
  2. 如果本机能通，公网不能通，就检查安全组、防火墙、云平台入站规则；
  3. 如果本机也不通，就看应用日志或错误日志；
  4. 检查服务是否只绑定了本地回环地址。

场景 4：端口被别的进程占用
例如：

  • ss -lntp | grep :80
  • lsof -i:80
    发现监听 80 的不是你想要的服务，而是别的程序。
    这通常说明：
    目标服务启动失败的根因可能是端口冲突。

下一步应当：

  1. 确认占用端口的是谁；
  2. 判断这个进程是不是本来就该存在；
  3. 如果不是预期进程，就停止它或修改目标服务的监听端口；
  4. 再重新启动目标服务并验证。

场景 5：进程和端口都正常，但业务仍然异常
例如：

  • nginx 在跑
  • 80 端口也在监听
  • 但 WordPress 页面报 502 / 403 / 数据库连接异常
    这时说明：
    基础服务层面可能没问题，问题更可能在应用链路内部。

下一步应当：

  1. 看错误日志；
  2. 检查后端 PHP-FPM / MySQL 是否正常；
  3. 检查目录权限、用户一致性、配置参数；
  4. 按“Web → PHP-FPM → MySQL”的链路逐层验证。

总结

总的来说，服务出问题就先看服务有没有在跑（ps 命令），接下来是看端口，服务有没有在正确的端口进行监听（ss -lntp | grep : 端口号、lsof -i: 端口），知道大概情况后，分类进行解决。

搭配上上一篇笔记，我现在的排障思路一般就是：
systemctl status 看服务状态 → ps aux 看进程 → ss 看端口监听 → lsof 看端口归属 → journalctl 看详细日志。

常用命令示例

  • ps 看进程是否存在、运行用户是谁
  • ss 看端口是否真的监听
  • lsof 看监听端口具体属于哪个进程
